## What it is

A focused engineering artifact, not a SaaS. The shipped pieces:

- **Dataset pipeline** — synthesize NDAs, MSAs, and licenses via GPT-4o with strict JSON-schema enforcement, then curate (length, language, schema, dedup), split stratified with an anti-contamination hash guard, and format into chat-template JSONL.
- **Training** — QLoRA (NF4 + LoRA, rank 16, alpha 32, `all_linear` targets) on Llama 3.1 8B. One shared `TrainingConfig` drives Unsloth on the 4090 and TRL's `SFTTrainer` on M1.
- **Eval** — three predictors behind one `Protocol` (fixture, local fine-tuned, GPT-4o) run over the same held-out test split, reporting JSON validity, per-field scores, token totals, latency, and API cost.
- **Cost model** — training dollars (GPU hours, synthesis API, eval API) plus a self-hosted $/1M-tokens estimate, reduced to a breakeven volume against continuing to call GPT-4o.
- **Publishing** — model-card renderer and Hugging Face Hub adapter upload.
- **CI** — GitHub Actions runs Ruff, mypy (strict), pytest, and commitlint on every PR. No GPU in CI; the paid run is a manual, methodology-driven step on RunPod.

## Why I built it

To prove I can fine-tune and ship an open model end to end, with a defensible cost story, rather than only calling a hosted API. The interesting question is not "can QLoRA fine-tune Llama 3.1 8B" (it can) but "does the fine-tune actually close the gap to a frontier model on a narrow task, and how long until the one-time training cost pays for itself?" Anvil is built to answer that with numbers that trace back to real inputs, and it pairs with Forge so the two cover training and serving as one portfolio story.

## How it works

### Data synthesis with an anti-contamination guard

The training set is synthetic contracts generated under a strict JSON schema, so every sample has a known ground-truth extraction. Before training, a guard hashes the normalized contract text of every sample and aborts the run if any document appears in more than one split. That removes the most common way a fine-tuning benchmark quietly inflates its own scores.

### QLoRA on a single 4090, rehearsed on M1

The same orchestrator that runs the paid RunPod job runs in a dry-run mode on a base-M1 MacBook against in-repo fixtures, end to end (`data-smoke → train-smoke → eval-smoke → cost`) with zero API spend. That rehearsal must pass before a paid GPU is rented, so config typos and parser bugs cost nothing instead of burning 4090 minutes. The full run is budget-capped near a $100 envelope, and synthesis aborts at a hard cap before any training spend.

### Three-way eval gated on JSON validity

Base, fine-tuned, and GPT-4o predictors answer the same prompts. Validity is the gate: an output that does not parse against the schema earns no field credit, because an F1 number on unparsable JSON is meaningless. The fine-tuned adapter is expected to reach near-total schema conformance where the base model fails on most prompts, but those figures stay marked illustrative until the paid run measures them.

### Cost and breakeven

The cost report sums GPU hours, synthesis API, and eval API into a one-time training cost, then computes a self-hosted inference $/1M tokens and the volume at which per-token savings repay training versus continuing to call GPT-4o. Forge's measured throughput is deliberately not quoted until Forge's own paid run lands; Anvil cites a published Llama 3.1 8B and 4090 source instead, so no number leans on an unmeasured assumption.

## Status

The full pipeline is landed and rehearsed end to end on M1 with zero API spend. The paid RunPod RTX 4090 run that replaces every illustrative figure with a measured one is queued; once it lands, a single commit swaps in the real validity, per-field, and cost numbers, and the final writeup follows.
