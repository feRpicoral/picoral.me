## What it is

A study, not a product. The shipped artifact is:

- A vLLM serving config with continuous batching, KV cache, and an OpenAI-compatible streaming API.
- An AWQ-INT4 quantization recipe.
- A benchmark harness measuring throughput, TTFT (p50/p95), and TPOT (p50/p95) under realistic concurrency.
- A quality eval wrapping `lm-evaluation-harness` (MMLU, GSM8K, HellaSwag) and reporting the full-precision-vs-quantized delta.
- A cost model producing $/1M tokens self-hosted vs. commercial APIs.
- Prometheus metrics + a Grafana dashboard for the serving leg.

## Why I built it

To demonstrate I can serve and optimize an open-source LLM end-to-end — not just call a hosted API. The interesting question isn't "can vLLM run Llama 3.1 8B" (yes, the docs cover that) — it's "what does the defensible $/1M-tokens number look like once you account for utilization, p95 latency, and quality regression from quantization?" That's what Forge measures.

## How it works

### Local development on commodity hardware

The full pipeline runs on a base-model M1 MacBook Pro against a tiny model (`Qwen/Qwen2.5-0.5B-Instruct`) before any paid GPU rental. The vLLM CPU build, the benchmark harness, the cost-model parsers, and the chart generator are all debugged locally for free. Only the final reproduction run consumes GPU minutes.

### Reproducibility

Methodology, hardware, model SHAs, exact vLLM version, and AWQ recipe live in `docs/methodology.md`. Benchmark output JSON is committed under `results/`. Anyone with the methodology doc and a rented GPU can reproduce the numbers — that's the point of the deliverable.

### CI

GitHub Actions runs Ruff (lint + format), mypy, and pytest on every PR and push to main. No GPU jobs in CI — benchmark reproduction is a manual, methodology-driven step on rented hardware.

## Status

Scaffolding. The serving layer, quantization recipe, benchmark harness, and cost model land across phases 1–8 of the implementation plan; the impact lead, full benchmark run, and charts land in phase 9.
