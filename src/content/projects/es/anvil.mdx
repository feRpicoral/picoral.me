## Qué es

Un artefacto de ingeniería enfocado, no un SaaS. Las piezas entregadas:

- **Pipeline de dataset** - generó 4,000 NDAs, MSAs y licencias sintéticos vía GPT-4o con aplicación estricta de JSON schema; curó 3,707 registros aceptados; los dividió en 2,966 de entrenamiento, 371 de validación y 370 de test.
- **Entrenamiento** - QLoRA (NF4 + LoRA, rank 16, alpha 32, targets `all_linear`) sobre Llama 3.1 8B vía Unsloth en un único RunPod A40. El run pago completó 558 pasos de optimizador en 8,259 segundos, aproximadamente 2h 18m.
- **Eval** - predictores base, fine-tuned y GPT-4o bajo un mismo protocolo, ejecutados sobre el mismo test split reservado de 370 ejemplos y puntuados en validez JSON más calidad de extracción por campo.
- **Modelo de costos** - costo medido del run de entrenamiento (API de síntesis, tiempo de GPU, eval con GPT-4o) más un modelo self-hosted de $/1M tokens y volumen de punto de equilibrio frente a seguir llamando a GPT-4o.
- **CI y ensayo** - el pipeline completo puede ensayarse en un M1 contra fixtures antes de rentar una GPU; el CI cubre parsing, schema, costos, gráficos, preflight, configuración de entrenamiento y utilidades de eval.

## Por qué lo construí

Para demostrar que puedo hacer fine-tuning y evaluar un modelo abierto de punta a punta, con una historia de costos defendible, en lugar de solo llamar a un API hosteado. La pregunta útil es si el fine-tuning cierra la brecha con un modelo frontier en una tarea estrecha de salida estructurada, y cuánto tiempo tarda el costo único de entrenamiento en amortizarse.

Anvil responde eso con números ligados a artefactos comprometidos: los conteos del dataset, el log de entrenamiento, el JSON de evaluación, las imágenes de gráficos y el reporte de costos son todos reproducibles desde el workflow del repo.

## Resultado medido

El modelo base falló el gate estricto de schema: 0% de JSON válido en el test split reservado de 370 casos. El adapter fine-tuned produjo 100% de JSON válido, levemente por encima del 99.73% de validez de GPT-4o en los mismos casos.

<ProjectImage src="/src/assets/projects/anvil/json-validity.png" alt="Gráfico de barras comparando la validez JSON de los predictores base, fine-tuned y GPT-4o." caption="La validez JSON es el gate: la salida no parseable no recibe crédito de campo." size="chart" />

En el puntaje de extracción por campo, el adapter fine-tuned alcanzó un macro score de 0.899 frente al 0.881 de GPT-4o. La tarea es estrecha y sintética, así que lo tomo como evidencia de que el adapter aprendió el schema objetivo y la distribución de campos, no como una afirmación amplia de que supera a GPT-4o.

## Cómo funciona

### Síntesis de datos con un guard anti-contaminación

El conjunto de entrenamiento son contratos sintéticos generados bajo un JSON schema estricto, por lo que cada muestra tiene una extracción ground-truth conocida. Antes del entrenamiento, un guard hashea el texto normalizado de cada muestra y aborta el run si algún documento aparece en más de un split. Eso elimina la forma más común en que un benchmark de fine-tuning infla silenciosamente sus propios puntajes.

### QLoRA en un único A40, ensayado en M1

El mismo orquestador que ejecuta el job pago en RunPod corre en modo dry-run en un MacBook M1 base contra fixtures del repo, de punta a punta (`data-smoke -> train-smoke -> eval-smoke -> cost`) con cero gasto en API. Ese ensayo debe pasar antes de rentar una GPU paga.

El run pago usó un RunPod A40 a $0.44/hr, entrenó por 3 épocas y terminó con train loss 0.346 y eval loss 0.328.

<ProjectImage src="/src/assets/projects/anvil/training-loss.png" alt="Gráfico de líneas del train loss y validation loss de Anvil a lo largo de 558 pasos de optimizador." caption="El entrenamiento convergió rápido y se mantuvo estable durante el run de 3 épocas en A40." size="chart" />

### Eval de tres vías condicionada a validez JSON

Los predictores base, fine-tuned y GPT-4o responden los mismos prompts. La validez es el gate: una salida que no parsea contra el schema no recibe crédito de campo, porque un número F1 sobre JSON no parseable no tiene sentido. Aquí es donde el modelo base colapsa y el fine-tune se vuelve útil.

### Costo y punto de equilibrio

El run completo costó $58.46: $54.00 por síntesis, $1.76 por tiempo de GPU en A40 y $2.70 por eval con GPT-4o. La línea de inferencia self-hosted está etiquetada intencionalmente como un supuesto configurado de serving en A40, no como un benchmark de serving medido en este proyecto.

<ProjectImage src="/src/assets/projects/anvil/cost-per-1m.png" alt="Gráfico de barras comparando el costo self-hosted configurado de Anvil en A40 por millón de tokens frente a GPT-4o y Claude Sonnet." caption="El costo self-hosted usa el supuesto configurado de throughput de serving en A40." size="chart" />

Con esos supuestos, el costo único del run alcanza el punto de equilibrio frente a GPT-4o a 0.80M tokens por mes en un horizonte de 12 meses.

<ProjectImage src="/src/assets/projects/anvil/breakeven.png" alt="Curva de punto de equilibrio comparando el costo acumulado del modelo fine-tuned frente a GPT-4o a lo largo de doce meses." caption="Volumen de punto de equilibrio: 0.80M tokens por mes frente a GPT-4o." size="chart" />
