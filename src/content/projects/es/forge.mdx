## Qué es

Un estudio, no un producto. El artefacto entregado es:

- Una configuración de serving en vLLM con continuous batching, KV cache y una API de streaming compatible con OpenAI.
- Una receta de cuantización AWQ-INT4.
- Un harness de benchmark que mide throughput, TTFT (p50/p95) y TPOT (p50/p95) bajo concurrencia realista.
- Una evaluación de calidad que envuelve `lm-evaluation-harness` (MMLU, GSM8K, HellaSwag) y reporta el delta entre precisión completa y cuantizado.
- Un modelo de costos que produce $/1M tokens autoalojado vs. APIs comerciales.
- Métricas de Prometheus + un dashboard de Grafana para la capa de serving.

## Por qué lo construí

Para demostrar que puedo servir y optimizar un LLM de código abierto de punta a punta — no solo llamar a una API hosteada. La pregunta interesante no es "¿puede vLLM correr Llama 3.1 8B?" (sí, la documentación lo cubre) — es "¿cómo queda el número defensible de $/1M tokens una vez que se consideran la utilización, la latencia p95 y la regresión de calidad por cuantización?" Eso es lo que Forge mide.

## Cómo funciona

### Desarrollo local en hardware de consumo

El pipeline completo corre en una MacBook Pro M1 base contra un modelo pequeño (`Qwen/Qwen2.5-0.5B-Instruct`) antes de rentar cualquier GPU paga. El build de vLLM para CPU, el harness de benchmark, los parsers del modelo de costos y el generador de gráficos se depuran todos localmente y de forma gratuita. Solo la corrida final de reproducción consume minutos de GPU.

### Reproducibilidad

La metodología, el hardware, los SHAs del modelo, la versión exacta de vLLM y la receta AWQ viven en `docs/methodology.md`. El JSON de salida del benchmark se commitea en `results/`. Cualquiera con el doc de metodología y una GPU rentada puede reproducir los números — ese es el punto del entregable.

### CI

GitHub Actions ejecuta Ruff (lint + format), mypy y pytest en cada PR y push a main. Sin jobs de GPU en CI — la reproducción del benchmark es un paso manual, guiado por la metodología, en hardware rentado.

## Estado

Scaffolding. La capa de serving, la receta de cuantización, el harness de benchmark y el modelo de costos se incorporan en las fases 1–8 del plan de implementación; el impact lead, la corrida completa de benchmark y los gráficos llegan en la fase 9.
