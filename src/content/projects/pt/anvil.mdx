## O que é

Um artefato de engenharia focado, não um SaaS. As peças entregues:

- **Pipeline de dataset** — sintetiza NDAs, MSAs e licenças via GPT-4o com enforcement estrito de JSON schema, depois faz curadoria (tamanho, idioma, schema, dedup), split estratificado com um guard de hash anti-contaminação, e formata em JSONL de chat-template.
- **Treinamento** — QLoRA (NF4 + LoRA, rank 16, alpha 32, targets `all_linear`) no Llama 3.1 8B. Um único `TrainingConfig` compartilhado aciona o Unsloth na 4090 e o `SFTTrainer` do TRL no M1.
- **Eval** — três preditores atrás de um único `Protocol` (fixture, fine-tuned local, GPT-4o) rodando sobre o mesmo split de teste reservado, reportando validade de JSON, scores por campo, total de tokens, latência e custo de API.
- **Modelo de custo** — dólares de treinamento (horas de GPU, API de síntese, API de eval) mais uma estimativa self-hosted de $/1M tokens, reduzida a um volume de breakeven contra continuar chamando o GPT-4o.
- **Publicação** — renderizador de model card e upload do adapter para o Hugging Face Hub.
- **CI** — GitHub Actions roda Ruff, mypy (strict), pytest e commitlint em todo PR. Sem GPU no CI; a execução paga é um passo manual, orientado por metodologia, no RunPod.

## Por que construí

Para provar que consigo fazer fine-tuning e entregar um modelo aberto de ponta a ponta, com uma história de custo defensável, em vez de só chamar uma API hospedada. A questão interessante não é "o QLoRA consegue fazer fine-tuning do Llama 3.1 8B" (consegue), mas "o fine-tuning realmente fecha a lacuna para um modelo de fronteira em uma tarefa específica, e quanto tempo até o custo único de treinamento se pagar?" O Anvil é construído para responder isso com números que remetem a inputs reais, e se combina com o Forge para que os dois cubram treinamento e serving como uma história única de portfólio.

## Como funciona

### Síntese de dados com guard anti-contaminação

O conjunto de treinamento é composto por contratos sintéticos gerados sob um JSON schema estrito, então cada amostra tem uma extração de ground-truth conhecida. Antes do treinamento, um guard faz hash do texto normalizado do contrato de cada amostra e aborta a execução se algum documento aparecer em mais de um split. Isso elimina a forma mais comum de um benchmark de fine-tuning inflar silenciosamente seus próprios scores.

### QLoRA em uma única 4090, ensaiado no M1

O mesmo orquestrador que roda o job pago no RunPod roda em modo dry-run em um MacBook M1 base contra fixtures do repositório, de ponta a ponta (`data-smoke → train-smoke → eval-smoke → cost`) com zero gasto em API. Esse ensaio precisa passar antes de alugar uma GPU paga, então erros de config e bugs de parser não custam nada em vez de queimar minutos de 4090. A execução completa tem um teto de orçamento próximo a $100, e a síntese aborta em um hard cap antes de qualquer gasto com treinamento.

### Eval três a três com gate de validade de JSON

Os preditores base, fine-tuned e GPT-4o respondem aos mesmos prompts. A validade é a porta de entrada: um output que não faz parse contra o schema não recebe crédito por campo, porque um número de F1 em JSON não parseável não tem sentido. Espera-se que o adapter fine-tuned alcance conformidade quase total com o schema onde o modelo base falha na maioria dos prompts, mas esses números ficam marcados como ilustrativos até a execução paga os medir.

### Custo e breakeven

O relatório de custo soma horas de GPU, API de síntese e API de eval em um custo único de treinamento, depois calcula um $/1M tokens de inferência self-hosted e o volume no qual a economia por token paga o treinamento versus continuar chamando o GPT-4o. O throughput medido do Forge é deliberadamente não citado até a execução paga do Forge chegar; o Anvil cita uma fonte publicada do Llama 3.1 8B e da 4090, para que nenhum número dependa de uma premissa não medida.

## Status

O pipeline completo está pronto e ensaiado de ponta a ponta no M1 com zero gasto em API. A execução paga no RunPod com RTX 4090, que substitui cada número ilustrativo por um medido, está na fila; quando chegar, um único commit troca os números reais de validade, por campo e de custo, e o writeup final vem em seguida.
