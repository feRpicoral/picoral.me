## O que é

Um artefato de engenharia focado, não um SaaS. As peças entregues:

- **Pipeline de dataset** - gerou 4.000 NDAs, MSAs e licenças sintéticos via GPT-4o com enforcement estrito de JSON schema; curou 3.707 registros aceitos; dividiu em 2.966 de treino, 371 de validação e 370 de teste.
- **Treinamento** - QLoRA (NF4 + LoRA, rank 16, alpha 32, targets `all_linear`) no Llama 3.1 8B via Unsloth em um único RunPod A40. O run pago completou 558 optimizer steps em 8.259 segundos, cerca de 2h 18m.
- **Eval** - preditores base, fine-tuned e GPT-4o sob um único protocolo, rodados no mesmo split de teste reservado de 370 exemplos e pontuados em validade JSON mais qualidade de extração por campo.
- **Modelo de custo** - custo medido do run de treinamento (API de síntese, tempo de GPU, eval do GPT-4o) mais um modelo self-hosted de $/1M tokens e volume de breakeven contra continuar chamando o GPT-4o.
- **CI e ensaio** - o pipeline completo pode ser ensaiado em um M1 contra fixtures antes de alugar uma GPU; o CI cobre parsing, schema, custo, gráficos, preflight, configuração de treinamento e utilitários de eval.

## Por que construí

Para provar que consigo fazer fine-tuning e avaliar um modelo aberto de ponta a ponta, com uma história de custo defensável, em vez de apenas chamar uma API hospedada. A questão útil é se o fine-tuning fecha a lacuna para um modelo de fronteira em uma tarefa estreita de saída estruturada, e quanto tempo leva para o custo único de treinamento se pagar.

O Anvil responde isso com números vinculados a artefatos comprometidos: as contagens do dataset, o log de treinamento, o JSON de avaliação, as imagens dos gráficos e o relatório de custo são todos reproduzíveis a partir do workflow do repositório.

## Resultado medido

O modelo base falhou na gate estrita de schema: 0% de JSON válido no split de teste reservado de 370 casos. O adapter fine-tuned produziu 100% de JSON válido, levemente acima dos 99,73% de validade do GPT-4o nos mesmos casos.

<ProjectImage src="/src/assets/projects/anvil/json-validity.png" alt="Gráfico de barras comparando a validade JSON dos preditores base, fine-tuned e GPT-4o." caption="A validade JSON é a gate: saídas não parseáveis não recebem crédito de campo." size="chart" />

No score de extração por campo, o adapter fine-tuned atingiu 0,899 de macro score contra 0,881 do GPT-4o. A tarefa é estreita e sintética, então trato isso como evidência de que o adapter aprendeu o schema alvo e a distribuição de campos, não como uma afirmação ampla de que supera o GPT-4o.

## Como funciona

### Síntese de dados com guarda anti-contaminação

O conjunto de treinamento é composto por contratos sintéticos gerados sob um JSON schema estrito, então cada amostra tem uma extração ground-truth conhecida. Antes do treinamento, uma guarda faz hash do texto normalizado de cada amostra e aborta o run se algum documento aparecer em mais de um split. Isso elimina a forma mais comum de um benchmark de fine-tuning inflar silenciosamente seus próprios scores.

### QLoRA em um único A40, ensaiado no M1

O mesmo orquestrador que roda o job pago no RunPod roda em modo dry-run em um MacBook M1 base contra fixtures do repositório, de ponta a ponta (`data-smoke -> train-smoke -> eval-smoke -> cost`) com zero gasto em API. Esse ensaio precisa passar antes de alugar uma GPU paga.

O run pago usou um RunPod A40 a $0,44/hr, treinou por 3 épocas e terminou com train loss 0,346 e eval loss 0,328.

<ProjectImage src="/src/assets/projects/anvil/training-loss.png" alt="Gráfico de linhas do train loss e validation loss do Anvil ao longo de 558 optimizer steps." caption="O treinamento convergiu rapidamente e se manteve estável durante o run de 3 épocas no A40." size="chart" />

### Eval em três vias com gate de validade JSON

Os preditores base, fine-tuned e GPT-4o respondem os mesmos prompts. A validade é a gate: uma saída que não parseia contra o schema não recebe crédito de campo, porque um número de F1 em JSON não parseável não tem sentido. É aqui que o modelo base colapsa e o fine-tuning se torna útil.

### Custo e breakeven

O run completo custou $58,46: $54,00 pela síntese, $1,76 pelo tempo de GPU no A40 e $2,70 pelo eval do GPT-4o. A linha de inferência self-hosted é intencionalmente rotulada como premissa configurada de serving no A40, não como um benchmark de serving medido neste projeto.

<ProjectImage src="/src/assets/projects/anvil/cost-per-1m.png" alt="Gráfico de barras comparando o custo por milhão de tokens do A40 self-hosted configurado no Anvil contra GPT-4o e Claude Sonnet." caption="O custo self-hosted usa a premissa configurada de throughput de serving no A40." size="chart" />

Com essas premissas, o custo único do run atinge breakeven contra o GPT-4o em 0,80M de tokens por mês ao longo de um horizonte de 12 meses.

<ProjectImage src="/src/assets/projects/anvil/breakeven.png" alt="Curva de breakeven comparando o custo acumulado do modelo fine-tuned contra o GPT-4o ao longo de doze meses." caption="Volume de breakeven: 0,80M de tokens por mês contra o GPT-4o." size="chart" />
